# MyTik 技术栈选型说明

## 1. 文档目的

本文档用于明确 MyTik 项目在第一阶段（微信小程序验证期）的技术栈、架构边界、实现原则与后续演进方向。  
如与其他默认偏好冲突，以项目 PRD 与本说明为准。

适用范围：

- 微信小程序首期开发
- 基于 `uni-app` 的跨端代码组织
- 本地优先、隐私优先、轻量闭环的产品实现

---

## 2. 结论摘要

第一阶段推荐采用以下技术栈：

- 框架：`uni-app`
- 语言：`TypeScript`
- 前端框架：`Vue 3`
- 构建工具：`Vite`
- 状态管理：`Pinia`
- 样式方案：`SCSS` + CSS 变量
- UI 方案：`uni-ui` 少量复用 + 项目自定义业务组件
- 本地存储：`uni.getStorageSync` / `uni.setStorageSync`
- 文件能力：微信小程序 `FileSystemManager`
- 媒体能力：`wx.chooseMedia`
- 生物识别：`uni.startSoterAuthentication`
- 工具库：`dayjs`、`nanoid`、`zod`（可选）
- 代码规范：`ESLint` + `Prettier`

第一阶段默认不引入业务后端，不接入云数据库，不上传业务视频与用户元数据。

---

## 3. 选型依据

根据当前 PRD，项目的核心特征如下：

- 产品首期目标端是微信小程序
- 核心价值是“本地闭环、绝对隐私、零业务数据采集”
- 核心能力集中在本地视频导入、沉浸式播放、分类管理、个人弹幕留存、备份恢复
- 当前版本明确为纯单机，不包含任何他人互动
- 第三阶段存在演进到 Android / iOS App 的明确需求

因此，技术选型必须满足以下条件：

- 优先适配微信小程序
- 对本地文件与原生能力调用足够友好
- 能以较低成本复用到后续 App 端
- 不默认绑定云端数据架构
- 能快速完成 MVP 并持续迭代
- 不依赖不稳定或小程序受限的隐私锁能力

`uni-app` 在这一阶段比纯原生小程序更适合，原因是：

- 首期可以稳定交付微信小程序
- 后续可以沿用同一套业务代码扩展到 App
- 对 Vue 生态和组件化开发支持成熟
- 对本地能力封装、页面组织、跨端维护更友好

---

## 4. 推荐技术栈

### 4.1 核心框架

- `uni-app`
- `Vue 3`
- `TypeScript`
- `Vite`

选择原因：

- `uni-app` 满足微信小程序首发与后续 App 演进诉求
- `Vue 3` 适合页面型应用、组件拆分和状态组织
- `TypeScript` 有利于约束本地数据结构、导入导出 JSON、播放器状态与恢复逻辑
- `Vite` 启动与构建效率高，工程体验优于旧版方案

### 4.2 状态管理

- `Pinia`

选择原因：

- 与 Vue 3 组合自然
- 适合组织用户设置、分类数据、播放状态、个人评论状态
- 比自行拼接全局状态更易维护

建议拆分为以下 store：

- `useUserStore`
- `useLibraryStore`
- `usePlayerStore`
- `useCommentStore`

### 4.3 UI 与组件策略

- 通用基础组件可少量使用 `uni-ui`
- 业务核心界面以自定义组件为主

原因：

- 本项目 UI 风格强，强调沉浸式、深色、玻璃态、密码锁、全屏视频与弹层反馈
- 现成 UI 库只能覆盖少量基础能力，核心页面最终仍需定制

建议原则：

- 只复用基础型组件能力，如弹层、按钮、基础输入、空状态
- 视频卡片、密码键盘、弹幕层、分类卡片、全屏播放器等均使用项目组件实现

### 4.4 样式方案

- `SCSS`
- CSS 变量（主题色、圆角、阴影、层级、动效时长）

原因：

- 便于统一深色主题与组件 token
- 便于后续做主题扩展与全局风格控制
- 比完全依赖原子类方案更适合小程序和高定制业务界面

不推荐在当前项目首期采用重度 `Tailwind CSS` 作为主样式方案，原因是：

- `uni-app` 小程序侧以原生样式与组件约束为主
- 当前项目更需要强控制的业务组件与局部复杂布局
- 首期目标是快速稳定交付，不是引入额外样式工程复杂度

### 4.5 本地存储与数据层

首期数据层推荐完全本地化：

- 元数据存储：`uni.getStorageSync` / `uni.setStorageSync`
- 视频文件持久化：微信 `FileSystemManager.saveFile`
- 文件列表与容量检查：`FileSystemManager.getSavedFileList`

本地存储内容包括：

- 用户设置
- 分类信息
- 视频元数据
- 点赞状态
- 播放统计
- 个人弹幕/评论数据
- 备份版本信息

建议增加统一的数据访问层：

- `services/storage/`
- `repositories/`

目标：

- 避免在页面和组件里直接散落 `setStorageSync`
- 所有 JSON 读写统一收口
- 为未来数据迁移和版本升级预留空间

### 4.6 小程序原生能力封装

建议封装以下能力模块：

- `media.ts`
  - 封装 `wx.chooseMedia`
- `file-system.ts`
  - 封装 `FileSystemManager`
- `share.ts`
  - 封装备份导出与分享
- `permission.ts`
  - 封装相册、存储、系统能力检测

原因：

- 降低页面层对平台 API 的直接依赖
- 便于后续兼容 App 端不同实现
- 便于统一错误处理与边界兜底

---

## 5. 第一阶段为什么不需要后端

根据当前 PRD，第一阶段不建议建设业务后端。

原因如下：

- 首期重点是验证“本地短视频收藏与播放闭环”
- PRD 明确强调零业务数据上传
- 视频、分类、弹幕、点赞、设置均可在本地闭环完成
- 备份恢复通过本地 JSON 文件即可完成
- 过早引入后端会增加复杂度、隐私负担和开发成本

因此，第一阶段默认不引入：

- 云数据库
- 用户账号系统
- 服务端视频管理
- 在线业务 API
- 云端评论或云端同步

---

## 6. 未来什么情况下需要后端

仅当未来产品方向发生显著变化时，再按“最小必要原则”增加后端能力。

### 6.1 需要完整后端的场景

- 账号体系
- 跨设备云同步
- 云端备份
- 订阅付费
- 运营消息推送
- 审核、风控、数据分析

在未进入上述阶段前，不建议预建设完整后端。

---

## 7. 不推荐的方案

### 7.1 首期不推荐引入 Supabase 或其他云数据库

原因：

- 与 PRD 的“零业务数据上传”原则冲突
- 会弱化首期产品定位
- 会显著增加认证、同步、数据迁移和隐私说明成本

### 7.2 首期不推荐重度依赖大型 UI 框架

如 `uView` 一类的大而全 UI 库，不建议作为核心依赖。

原因：

- 本项目不是标准信息流表单应用
- 强业务交互页面最终需要高度自定义
- 过度依赖会增加样式覆盖成本与包袱

### 7.3 首期不推荐先做复杂跨端抽象

原因：

- 当前最重要的是先把微信小程序 MVP 跑通
- 过早抽象 App 端差异，会拖慢交付速度
- 与其预抽象，不如先把平台能力封装边界做好

---

## 8. 推荐目录结构

建议采用如下结构：

```text
pages/
  library/
  home/
  settings/
  lock/
components/
  business/
  common/
stores/
services/
  platform/
  storage/
repositories/
utils/
types/
theme/
constants/
```

说明：

- `pages/`：页面入口与 tab 页面
- `components/business/`：播放器、分类卡片、评论输入、弹幕层等业务组件
- `components/common/`：通用基础组件
- `stores/`：全局状态
- `services/platform/`：平台能力封装
- `services/storage/`：本地存储读写与迁移
- `repositories/`：面向业务的数据访问层
- `types/`：统一维护 PRD 对应的数据结构
- `theme/`：主题变量与样式 token

---

## 9. 数据结构建议

首期应严格围绕 PRD 中的本地 JSON 结构建模，并补充以下约束：

- 所有核心对象必须带 `id`
- 所有核心对象必须带 `createdAt`
- 建议增加 `updatedAt`
- 备份文件必须带 `schemaVersion`
- 导入时必须做结构校验
- 未来迁移必须保留版本升级入口

建议定义：

- `UserSettings`
- `Category`
- `VideoAsset`
- `CommentItem`
- `BackupPayload`

如无特殊性能瓶颈，优先保持结构清晰，不要过早做复杂索引优化。

---

## 10. 工程规范建议

### 10.1 代码规范

- 使用 `ESLint`
- 使用 `Prettier`
- 使用 `TypeScript` 严格模式

### 10.2 命名规范

- 组件名使用 `PascalCase`
- composables / stores 使用 `useXxx`
- 平台能力封装统一放在 `services/platform`
- 存储 key 统一集中管理，避免硬编码

### 10.3 实现原则

- 优先最小可运行版本
- 优先可验证闭环
- 优先本地能力稳定性
- 避免无关重构
- 避免为了未来可能需求而过度设计

---

## 11. 首期 MVP 推荐实现范围

建议首期优先完成以下能力：

1. 启动锁屏与本地密码
2. 视频导入与本地持久化
3. 分类管理与封面展示
4. Home 页沉浸式全屏滑动播放
5. 点赞、播放次数、观看时长统计
6. 本地弹幕/评论展示、新增与回显
7. JSON 备份导出
8. JSON 导入恢复
9. 基于轻量特征的断链修复

暂缓实现：

- 隐私锁
- 生物识别解锁
- 完整云同步
- 账号与多端登录

---

## 12. 最终选型结论

MyTik 第一阶段微信小程序项目，推荐采用：

`uni-app + Vue 3 + TypeScript + Vite + Pinia + SCSS + uni-ui（少量）+ 微信原生文件/媒体能力`

并坚持以下架构原则：

- 首期无后端
- 本地优先
- 隐私优先
- 以微信小程序 MVP 为第一目标
- 保留后续扩展到 App 端的代码组织能力

该方案是在当前 PRD 约束下，兼顾产品速度、实现复杂度、后续演进与维护成本的最优平衡方案。
